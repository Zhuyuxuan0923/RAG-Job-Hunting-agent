## 1. 核心概念速览

| 概念 | 一句话解释 | 类比 |
|---|---|---|
| **Docker** | 把应用和它需要的所有东西打包在一起的技术 | 集装箱——里面有什么、怎么装的，外面不用管 |
| **镜像（Image）** | 打包好的只读模板，包含代码+环境+依赖 | 菜谱 + 食材包，拿来就能做菜 |
| **容器（Container）** | 镜像跑起来之后的实例 | 按菜谱做出来的一盘菜 |
| **Docker Compose** | 编排多个容器的工具，一条命令启动/停止全部 | 总厨师长，同时调度多个灶台 |
| **部署** | 让应用在服务器上跑起来，用户能访问 | 把菜端上桌 |
| **运维** | 部署之后持续看护，保证服务不挂 | 盯着菜别凉了，盘子别碎了 |
| **反向代理** | 用户请求先到 Nginx，Nginx 再转发给后端 | 前台接待——客人说找谁，前台领进去 |
| **SSL/HTTPS** | 给网站加把锁，数据加密传输 | 把明信片换成密封信封 |

## 2. 容器化：把项目装进箱子

### 2.1 为什么要容器化？

**没有 Docker 时**：
```
开发："我电脑上能跑啊！"
运维："服务器上报错，少了个系统库。"
开发："你 Python 版本多少？"
运维："3.10。"
开发："我用的是 3.14，那个特性 3.10 不支持..."
```

**有 Docker 后**：
```
开发：我用 Docker 打包好了，你直接 docker compose up。
运维：跑起来了。
```

Docker 解决的三个核心问题：
1. **环境一致**：开发、测试、生产环境完全一样
2. **依赖隔离**：不污染宿主机，不跟其他项目冲突
3. **一键部署**：新机器上只要有 Docker，一条命令就能跑

### 2.2 Docker 工作流程

```
Dockerfile             docker build          docker run
(配方说明书)    ---→     镜像（模板）    ---→     容器（运行实例）
```

举个例子，本项目的后端 Dockerfile：

```dockerfile
FROM python:3.14-slim           # 1. 基于官方 Python 镜像
WORKDIR /app                     # 2. 设定工作目录
RUN pip install poetry           # 3. 装包管理器
COPY pyproject.toml poetry.lock ./  # 4. 复制依赖清单
RUN poetry install               # 5. 安装依赖
COPY src/ ./src/                 # 6. 复制源代码
CMD ["uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]  # 7. 启动命令
```

每一步都是一层"缓存"，改代码只重建最后一层，前面不动，所以重复构建很快。

### 2.3 多阶段构建

注意我们的后端 Dockerfile 实际有两段：

```dockerfile
# 阶段一：builder — 安装依赖（带编译工具，体积大）
FROM python:3.14-slim AS builder
RUN poetry install ...

# 阶段二：runtime — 只复制装好的包（干净，体积小）
FROM python:3.14-slim
COPY --from=builder /usr/local/lib/python3.14/site-packages ...
COPY src/ ./src/
```

**为什么要两段？** 第一阶段安装依赖需要 gcc、dev 头文件等编译工具，`poetry install` 完成后这些工具就没用了。第二阶段从一个全新的干净镜像开始，只复制最终需要的文件。最终镜像能小 30%~50%。

## 3. 本项目 Docker 文件详解

### 3.1 文件清单

```
项目根目录/
├── docker-compose.yml       ← 生产部署编排
├── docker-compose.dev.yml   ← 开发模式编排（热重载）
├── deploy.sh / deploy.bat   ← 一键启动脚本
├── backend/
│   ├── Dockerfile            ← 后端镜像配方
│   └── .dockerignore         ← 构建时忽略的文件
└── frontend/
    ├── Dockerfile            ← 前端镜像配方
    ├── nginx.conf            ← Nginx 配置（反向代理 + 静态文件）
    └── .dockerignore         ← 构建时忽略的文件
```

### 3.2 docker-compose.yml 逐段解释

```yaml
services:
  backend:                          # 第一个服务：后端
    build: ./backend                # 用 backend/ 目录的 Dockerfile 构建
    ports:
      - "8000:8000"                 # 本机8000 → 容器8000
    env_file: .env                  # 从 .env 文件读 API Key
    environment:
      - CHROMA_PERSIST_DIR=/app/chroma_data  # 向量数据库存哪
    volumes:
      - chroma_data:/app/chroma_data  # 数据卷：删容器不丢数据
    healthcheck:                    # 健康检查
      test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://localhost:8000/api/health')"]
      interval: 30s                 # 每30秒查一次
      retries: 3                    # 连续3次失败判定为不健康

  frontend:                         # 第二个服务：前端
    build: ./frontend
    ports:
      - "80:80"                     # 本机80 → 容器80（浏览器默认端口，访问时不用写端口号）
    depends_on:                     # 等后端就绪再启动
      backend:
        condition: service_healthy  # 等健康检查通过

volumes:
  chroma_data:                      # 声明持久化存储卷
    name: rag-chroma-data           # 卷的名字
```

### 3.3 nginx.conf — 反向代理

用户访问 `http://localhost`，Nginx 充当"接线员"：

```
浏览器                           Nginx                       后端
   │                               │                          │
   ├─ GET / ───────────────────→   │                          │
   │                               ├─ 返回 index.html    （静态文件） 
   │                               │                          │
   ├─ GET /api/match/abc/stream ─→ │                          │
   │                               ├─ proxy_pass ───────────→ │
   │                               │                    (Python 处理)
   │                               │  ← SSE流式数据 ─────────┤
   │  ← 实时显示分析进度 ─────────┤  (proxy_buffering off    │
   │                               保证数据不堆积，逐条推送)  │
```

关键的 SSE 配置：

```nginx
location /api/ {
    proxy_pass http://backend:8000;
    proxy_buffering off;    # 关键：关掉缓冲，AI 回答是流式输出的
    proxy_cache off;         # 不缓存
    proxy_read_timeout 300s; # 超时设为5分钟（AI 分析可能要等）
}
```

### 3.4 docker-compose.dev.yml — 开发模式

跟生产模式的核心区别：

| | 生产 | 开发 |
|---|---|---|
| 后端 | 代码 COPY 进镜像，不改 | 本机目录 **映射** 进容器，改了自动生效 |
| 前端 | 编译后由 Nginx 提供 | Vite dev server，热重载 |
| 启动 | `docker compose up -d` | `docker compose -f docker-compose.dev.yml up -d` |

开发模式的 volumes 配置：

```yaml
backend:
  volumes:
    - ./backend/src:/app/src    # 本机 backend/src 目录 → 容器 /app/src
                                 # 覆盖掉镜像里原有的代码
```

所以你在 VS Code 里改 `backend/src/api/interview.py`，容器里的文件也同步变了，uvicorn `--reload` 自动重启。

## 4. 本地部署实操

### 4.1 环境要求

唯一的前置条件：**安装 Docker Desktop**。

- Windows: https://docs.docker.com/desktop/setup/install/windows-install/
- Mac: https://docs.docker.com/desktop/setup/install/mac-install/

装好之后确保 Docker Desktop 在运行（任务栏能看到鲸鱼图标）。

### 4.2 部署步骤

**第一步：准备 .env**

```bash
cp .env.example .env
```

编辑 `.env`，填入你的 API Key：

```ini
DEEPSEEK_API_KEY=sk-你的DeepSeek_API_Key
TAVILY_API_KEY=tvly-你的Tavily_API_Key
```

> 其他配置项都有默认值，可以不动。

**第二步：启动**

```bash
docker compose up -d --build
```

`-d` = 后台运行，`--build` = 强制重新构建镜像。

你会看到类似输出：

```
[+] Building 45.2s (23/23) FINISHED
[+] Running 3/3
 ✔ Network agenticrag_default    Created
 ✔ Container rag-backend         Healthy
 ✔ Container rag-frontend        Started
```

**第三步：验证**

打开浏览器访问 `http://localhost`，应该能看到首页。

```bash
# 查看运行状态
docker compose ps

# 查看日志
docker compose logs -f
```

### 4.3 常用命令速查

```bash
docker compose up -d --build    # 启动（重新构建）
docker compose up -d            # 启动（不重新构建）
docker compose down             # 停止并删除容器
docker compose down -v          # 停止 + 删除数据卷（ChromaDB数据会丢）
docker compose restart          # 重启
docker compose logs -f backend  # 只看后端日志
docker compose logs -f --tail=100 # 只看最近100行
docker compose ps               # 查看运行状态
docker compose exec backend bash  # 进入后端容器调试
```

### 4.4 更新代码后重新部署

```bash
git pull                       # 拉取最新代码
docker compose up -d --build   # 重新构建并重启
```

我们用的命名卷 `rag-chroma-data` 不会被删除，ChromaDB 里的数据继续保留。

## 5. 版本控制：给镜像打标签

### 5.1 为什么需要版本控制？

部署容器跟直接用代码的区别在于：你部署的是一个**镜像**，不是源码文件。

但镜像有个问题——如果你每次都 `docker compose build`，旧的镜像就被覆盖了，叫 `latest`：

```
rag-backend:latest    ← 只存了最新的一份
```

这有什么问题？

- **不知道线上跑的是哪版代码**：`latest` 可能是上周的，也可能是今天的
- **没法回滚**：旧镜像被覆盖了，想回去只能从 Git 翻历史再重新构建（慢）
- **不方便定位 Bug**：用户报"昨天还能用，今天不行了"，但你不知道昨天跑的是哪个版本

解决方式很简单：**用 Git tag 作为镜像 tag**。

### 5.2 版本号约定

我们采用 `v主版本.次版本.修订号` 的三位格式：

| 场景 | 版本号 | 例子 |
|---|---|---|
| 修 Bug | 改第三位 | v1.0.0 → v1.0.1 |
| 加新功能 | 改第二位 | v1.0.1 → v1.1.0 |
| 大改/不兼容 | 改第一位 | v1.1.0 → v2.0.0 |

不需要严格遵守，刚开始用 `v1.0.0`、`v1.0.1` 这样递增就行。

### 5.3 项目已集成的机制

`docker-compose.yml` 中已经配置了：

```yaml
services:
  backend:
    image: rag-backend:${TAG:-latest}    # 从环境变量 TAG 读版本号，没设就用 latest
  frontend:
    image: rag-frontend:${TAG:-latest}
```

`deploy.sh` 会自动检测当前 Git 状态：

```bash
# 如果当前 commit 有 tag，就用 tag 作为版本号
# 如果没有 tag，就用 latest
TAG=$(git describe --tags --exact-match 2>/dev/null || echo "latest")
```

### 5.4 日常开发（不管版本）

```bash
# 直接在 VS Code 里改代码，开发模式热重载
docker compose -f docker-compose.dev.yml up -d
```

没有 Git tag 时，镜像自动标记为 `latest`，不需要操心版本号。

### 5.5 发版上线

```bash
# 1. 确保代码已提交
git add -A
git commit -m "feat: 优化面试界面"

# 2. 打版本标签
git tag v1.0.0
git push origin v1.0.0

# 3. 用这个版本号构建并部署
./deploy.sh
# 或者手动指定：
# TAG=v1.0.0 docker compose up -d --build
```

构建完成后镜像库的状态：

```
rag-backend:v1.0.0    ← 版本化镜像，永久保留
rag-backend:latest    ← 指向最新版本，每次构建会覆盖
rag-frontend:v1.0.0
rag-frontend:latest
```

### 5.6 后续更新

```bash
# 开发 → 提交 → 打新 tag → 部署
git add -A && git commit -m "fix: 修复第一题不显示"
git tag v1.0.1 && git push origin v1.0.1
TAG=v1.0.1 docker compose up -d --build

# 现在镜像库里有两版：
# rag-backend:v1.0.1    ← 最新
# rag-backend:v1.0.0    ← 上一版，还在
```

### 5.7 一键回滚

```bash
# 线上 v1.0.1 出了问题，切回 v1.0.0
TAG=v1.0.0 docker compose up -d

# 不需要重新构建！旧镜像还在本地，秒级切换
```

这就是镜像不打 `latest` 的最大好处。

### 5.8 查当前在跑哪个版本

```bash
docker compose images
# 输出：
# Container       Image                   Tag
# rag-backend     rag-backend             v1.0.1
# rag-frontend    rag-frontend            v1.0.1
```

### 5.9 完整流程一张图

```
开发机                           服务器
  │                                │
  │ git tag v1.0.0                 │
  │ docker build -t rag-backend:v1.0.0  │
  │                                │
  │         ──── 推送 v1.0.0 ────→  │
  │                                │ docker compose up v1.0.0
  │                                │ → 上线成功
  │                                │
  │              ...两周后...       │
  │                                │
  │ git tag v1.1.0                 │
  │         ──── 推送 v1.1.0 ────→  │ docker compose up v1.1.0
  │                                │ → 更新上线
  │                                │
  │                                │ 😱 用户报 Bug
  │                                │ docker compose up v1.0.0
  │                                │ → 秒级回滚，用户无感
  │                                │
  │         ←── 查日志定位 Bug ───│
  │ git fix → v1.1.1               │
  │         ──── 推送 v1.1.1 ────→  │ docker compose up v1.1.1
  │                                │ → 修复上线
```

### 5.10 清理旧镜像

时间长了本地会积累很多旧版本镜像：

```bash
# 查看占用
docker images | grep rag-

# 删掉半年以上的旧版本
docker image prune -a --filter "until=4320h"

# 或手动删指定版本
docker rmi rag-backend:v1.0.0 rag-frontend:v1.0.0
```

> 注意：删掉的版本就没办法秒级回滚了（得重新从 Git 构建）。一般保留最近 3-5 个版本就够。

## 6. 上线：部署到云服务器

### 5.1 选服务器

推荐入门方案：

| 云厂商 | 推荐产品 | 最低配置 | 月费参考 |
|---|---|---|---|
| **阿里云** | ECS 轻量应用服务器 | 2核2G | ¥50-80 |
| **腾讯云** | 轻量应用服务器 | 2核2G | ¥50-80 |
| **AWS** | Lightsail | 2核2G | $20 |
| **DigitalOcean** | Droplet | 2核2G | $18 |

> 本项目需要至少 **2GB 内存**（ChromaDB 向量检索 + DeepSeek 模型调用需要内存）。

操作系统选 **Ubuntu 24.04 LTS**。

### 5.2 服务器初始化

SSH 连上服务器后：

```bash
# 1. 更新系统
sudo apt update && sudo apt upgrade -y

# 2. 安装 Docker
curl -fsSL https://get.docker.com | sudo bash
sudo usermod -aG docker $USER  # 免 sudo 使用 docker

# 3. 退出重新 SSH 登录，使权限生效
exit

# 4. 验证安装
docker --version
docker compose version
```

### 5.3 拉取代码并启动

```bash
# 克隆项目
git clone https://github.com/你的用户名/你的仓库名.git
cd 你的项目目录

# 配置环境变量
cp .env.example .env
vim .env   # 填入 API Key

# 启动
docker compose up -d --build
```

### 5.4 配置域名 + HTTPS

**有域名的话：**

```bash
# 1. 在域名 DNS 控制台添加 A 记录，指向服务器 IP
#    例如：rag.yourdomain.com → 你的服务器IP

# 2. 安装 Nginx + Certbot（免费 SSL 证书）
sudo apt install -y nginx certbot python3-certbot-nginx

# 3. 配置 Nginx 作为最外层反向代理
sudo vim /etc/nginx/sites-available/rag
```

```nginx
# /etc/nginx/sites-available/rag
server {
    listen 80;
    server_name rag.yourdomain.com;  # 换成你的域名

    location / {
        proxy_pass http://localhost:80;  # 转给 Docker 里的前端
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /api/ {
        proxy_pass http://localhost:8000;
        proxy_buffering off;          # SSE 流式
        proxy_read_timeout 300s;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

```bash
# 4. 启用配置 + 申请 SSL 证书
sudo ln -s /etc/nginx/sites-available/rag /etc/nginx/sites-enabled/
sudo rm /etc/nginx/sites-enabled/default   # 删掉默认站点
sudo nginx -t                               # 检查配置语法
sudo systemctl reload nginx

# 5. 申请免费 HTTPS 证书（Let's Encrypt）
sudo certbot --nginx -d rag.yourdomain.com

# 6. 设置证书自动续期
sudo systemctl enable certbot.timer
```

完成后访问 `https://rag.yourdomain.com`。

**没有域名、直接用 IP 访问的话**：目前 docker-compose 已经把前端暴露在 80 端口，服务器防火墙放行 80 端口就能直接 `http://你的IP` 访问。但强烈建议至少配个域名+HTTPS，因为 HTTP 明文传输不安全。

### 5.5 开放防火墙端口

```bash
# 如果服务器用的是 ufw 防火墙
sudo ufw allow 22    # SSH
sudo ufw allow 80    # HTTP
sudo ufw allow 443   # HTTPS
sudo ufw enable

# ⚠️ 云厂商控制台一般还有一层安全组防火墙，需要去控制台页面放行 80、443
```

## 7. 运维：上线之后的事

### 6.1 必须关注的三个指标

| 指标 | 意思 | 怎么查 |
|---|---|---|
| **服务是否活着** | 容器在不在跑 | `docker compose ps` |
| **资源够不够** | CPU/内存/磁盘 | `docker stats` |
| **有没有报错** | 日志里的异常 | `docker compose logs -f` |

### 6.2 日常巡检

```bash
# 1. 看状态——容器在跑就问题不大
docker compose ps
# 输出应该都显示 Up / healthy

# 2. 看资源——内存超过 90% 就要警惕
docker stats --no-stream

# 3. 看硬盘——满了会出各种怪问题
df -h
# 重点看 /var/lib/docker 和挂载的 chroma 数据卷

# 4. 看错误日志
docker compose logs --tail=50 | grep -i error
```

### 6.3 自动重启

`docker-compose.yml` 里已配置 `restart: unless-stopped`，意味着：
- 容器挂了 → Docker 自动重启
- 服务器重启 → 容器自动恢复

加上健康检查，如果后端连续 3 次 `/api/health` 不响应，Docker 会标记它为 unhealthy（虽然不会自动重启，但能让你快速发现）。

### 6.4 日志管理

Docker 默认日志会无限增长，长时间运行可能占满磁盘：

```bash
# 查看日志占用
docker system df

# 清理旧日志（建议加到 crontab 定时执行）
docker compose logs --tail=1000 > /tmp/rag-logs-$(date +%Y%m%d).log
docker system prune -f
```

或者在 `docker-compose.yml` 中限制日志大小：

```yaml
services:
  backend:
    logging:
      driver: "json-file"
      options:
        max-size: "50m"       # 单个日志文件最大 50MB
        max-file: "3"         # 最多保留3个文件
```

### 6.5 更新部署流程

```bash
# 1. 进入项目目录
cd /path/to/project

# 2. 拉取最新代码
git pull

# 3. 重新构建并重启（不用担心，chroma_data 卷不会丢）

# 4. 清理旧镜像（积累多了占磁盘）

```

### 6.6 备份与恢复

**需要备份什么？**

| 数据 | 位置 | 重要性 |
|---|---|---|
| 代码 | Git 仓库 | 不重要（可以重新 clone） |
| 配置 `.env` | 项目根目录 | **重要**（API Key，丢了要重新申请） |
| ChromaDB 向量数据 | Docker 命名卷 `rag-chroma-data` | 一般（可以重新解析文档重建索引） |

**备份脚本**：

```bash
#!/usr/bin/env bash
# backup.sh — 建议加到 crontab 每周跑一次

BACKUP_DIR=/home/ubuntu/backups
DATE=$(date +%Y%m%d)
mkdir -p $BACKUP_DIR

# 备份 .env
cp /path/to/project/.env $BACKUP_DIR/env-$DATE.bak

# 备份 ChromaDB 数据
docker run --rm -v rag-chroma-data:/data -v $BACKUP_DIR:/backup alpine tar czf /backup/chroma-$DATE.tar.gz -C /data .
```

## 8. 常见问题排查

### 7.1 容器起不来

```bash
# 看详细错误
docker compose logs backend

# 常见原因：
# - .env 文件不存在或 API Key 没填
# - 端口被占用（8000 或 80 端口有其他程序）
# - Docker Desktop 没运行
```

### 7.2 页面打开空白

```
F12 → Console 看有没有报错
F12 → Network 看 /api/ 请求是否 502 或 CORS 错误
```

### 7.3 AI 分析很久不出结果

```bash
# 检查后端日志，看卡在哪一步
docker compose logs -f backend

# 常见原因：
# - DeepSeek API 限流或欠费
# - Tavily 搜索超时
# - SSE 连接被代理中断（检查 nginx proxy_read_timeout）
```

### 7.4 磁盘空间不足

```bash
# 查看什么占空间
docker system df

# 清理无用数据
docker system prune -a   # 删除所有未使用的镜像、容器、网络
docker volume prune       # 删除未使用的卷（谨慎！会删数据）
```

### 7.5 改完代码怎么更新线上

```bash
git pull                              # 拉代码
docker compose up -d --build          # 重建重启
docker compose ps                     # 确认正常
```

> 如果前端改了什么但页面没变：可能是浏览器缓存，Ctrl+Shift+R 强制刷新。

## 总结：你只需要记住这几条命令

```bash
# 部署
docker compose up -d --build

# 看状态
docker compose ps

# 看日志
docker compose logs -f

# 重启
docker compose restart

# 停止
docker compose down

# 更新
git pull && docker compose up -d --build
```

遇到问题，先看日志：`docker compose logs -f`，绝大多数问题都能从日志里找到原因。
